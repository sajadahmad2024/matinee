## 🧱 1. Storage Class: `local-path-expandable`

### 📌 Purpose

This custom `StorageClass` enables **dynamic provisioning of local PersistentVolumes** in your k3s cluster. It uses the **`rancher.io/local-path` provisioner**, which is a lightweight local-path storage solution bundled with k3s.

The key enhancement here is the inclusion of:

```yaml
allowVolumeExpansion: true
```

This makes it possible to expand PVCs (Persistent Volume Claims) **after deployment**, which is not supported by the default local-path `StorageClass`.

---

### 📄 YAML Breakdown

```yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: local-path-expandable
  namespace: monitoring
provisioner: rancher.io/local-path
reclaimPolicy: Retain
volumeBindingMode: WaitForFirstConsumer
allowVolumeExpansion: true

```

### Key Fields Explained

| Field | Description |
| --- | --- |
| `provisioner: rancher.io/local-path` | Uses the built-in dynamic volume provisioner for k3s. |
| `reclaimPolicy: Retain` | Retains the underlying data when the PVC is deleted. |
| `volumeBindingMode: WaitForFirstConsumer` | Waits for a pod to be scheduled before binding a volume. Useful to match node-local paths. |
| `allowVolumeExpansion: true` | Allows resizing of PVCs (manual step required) even after they are bound. |

---

### ⚙️ How It Works

- When a PVC specifies `storageClassName: local-path-expandable`, k3s provisions a directory under `/opt/local-path-provisioner/` (by default).
- The volume is created only when a pod requiring the PVC is scheduled.
- With `allowVolumeExpansion: true`, you can later **patch the PVC's storage request**, and then **trigger a pod restart** to have the resized volume reflected.

---

### 🛠 Example PVC Using It

```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: prometheus-pvc
  namespace: monitoring
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 5Gi
  storageClassName: local-path-expandable
```

---

### 📏 Expanding a PVC (Step-by-Step)

1. **Patch the PVC**:
    
    ```bash
    kubectl patch pvc prometheus-pvc -n monitoring \
      -p '{"spec": {"resources": {"requests": {"storage": "10Gi"}}}}'
    ```
    
2. **Ensure the volume supports expansion**:
    
    The underlying file system (typically `ext4` or `xfs`) **must** support resizing. Also, ensure no processes are actively locking the mount.
    
3. **Restart the consuming pod** (so the new mount is applied):
    
    ```bash
    kubectl delete pod <pod-name> -n monitoring
    ```
    
4. **Verify**:
    
    ```bash
    kubectl get pvc prometheus-pvc -n monitoring
    ```
