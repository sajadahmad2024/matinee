## 5. Alertmanager Setup

**Alertmanager** is used to **handle alerts sent by Prometheus**, manage grouping, inhibition, silencing, and sending notifications through external services like **Slack**.

This configuration enables **Slack-based alert notifications** for your Kubernetes monitoring stack.


---

### ⚙️ Use Case & Behavior

| Component | Role |
| --- | --- |
| **ConfigMap** | Contains Alertmanager config with Slack integration logic. |
| **Deployment** | Runs a single Alertmanager instance. Binds config via a volume mount. |
| **Service** | Exposes Alertmanager internally in the cluster on port `9093`. |

---

### 🔧 Slack Integration Setup

> Before applying this configuration:
> 
- 🔁 Replace the placeholder:
    
    ```yaml
    api_url: 'https://hooks.slack.com/services/YOUR/SLACK/WEBHOOK'
    channel: '#alerts'
    ```
    
    with your actual Slack Incoming Webhook URL and preferred channel.
    
- You can create a Slack webhook in a specific workspace and channel, then paste it here.

---

### 📢 Alert Fields Explanation

The Slack notification includes:

| Field | Source | Description |
| --- | --- | --- |
| `alertname` | `.CommonLabels.alertname` | Name of the alert (e.g., `HighMemoryUsage`) |
| `summary` | `.CommonAnnotations.summary` | Short summary of the alert |
| `description` | `.CommonAnnotations.description` | Detailed alert info |
| `dashboard` | `.CommonAnnotations.dashboard` | Link to Grafana dashboard (if provided) |
| `severity` | `.CommonLabels.severity` | Alert severity: `critical`, `warning`, etc. |

---

### 📍 Node Placement Logic

This deployment is **restricted to a specific node** labeled with:

```yaml
node-role: monitoring
```

and tolerates this taint:

```yaml
monitoring-only=true:NoSchedule
```

> Ensure your monitoring node is tainted and labeled accordingly:
> 

```bash
kubectl label node monitoring node-role=monitoring
kubectl taint node monitoring monitoring-only=true:NoSchedule
```
