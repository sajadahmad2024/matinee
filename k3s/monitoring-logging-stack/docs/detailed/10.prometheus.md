## Prometheus Setup Documentation

### Namespace

- **Namespace Used:** `monitoring`
- All Prometheus-related resources are deployed within this namespace to isolate monitoring components.

---

## Alert Rules Config: `prometheus-alert-rules` ConfigMap

This ConfigMap includes predefined alerting rules and supports custom rule additions.

### Existing Rule Files:

- `node.rules`: Alerts on system-level resource usage.
- `blackbox-exporter.rules`: Alerts on Blackbox-exported endpoint availability.
- `custom-app.rules`: Alerts on custom application-level availability.

---

### ➕ Adding New Alert Rules

You can **extend the alert rules** by adding more `.rules` blocks inside the `prometheus-alert-rules` ConfigMap.

### Example: Adding a Custom Alert Rule

You can define a new block like the following inside `alert-rules-configmap.yaml` under the `data:` section.

```yaml
yaml
CopyEdit
  custom-app.rules: |
    groups:
    - name: endpoint_availability_alerts
      rules:
        # Alert for the <deployment-name-from-manifest's-metadata> service endpoint
        - alert: BackendProdEndpointDown
          expr: |
            absent (kube_endpoint_address{namespace="<namespace>", endpoint="<app-name>"}) == 1
          for: 5m
          labels:
            team: devops
            severity: critical
          annotations:
            summary: "Backend service endpoint is down (endpoint: {{ $labels.endpoint }}, namespace: {{ $labels.namespace }})"
            description: |
              The <app-name> endpoint in the <namespace> namespace has no active addresses.
              This may indicate that the service is not reachable or all its pods are down.
              Investigate immediately.
            dashboard: "<grafana.com>"

```

> 📝 Note: The filename key (like custom-app.rules) must follow the <rulename>.rules format, and all files will be automatically picked up by Prometheus via the /etc/prometheus/rules/*.rules rule path.
>

## Prometheus Configuration: `prometheus-config` ConfigMap

This ConfigMap contains the `prometheus.yml` configuration and is responsible for all scrape and alert configurations.

---

### Global Config

```yaml
global:
  scrape_interval: 15s
  evaluation_interval: 15s
```

---

### Alertmanager Config

```yaml
alerting:
  alertmanagers:
    - static_configs:
        - targets: ["alertmanager.monitoring.svc:9093"]
```

---

### Rule Files

```yaml
yaml
CopyEdit
rule_files:
  - /etc/prometheus/rules/*.rules
```

All `.rules` files defined in the `prometheus-alert-rules` ConfigMap will be loaded from this directory.

### Scrape Configs

### 🟢 Predefined Scrape Jobs:

1. **kube-state-metrics**

```yaml
- job_name: 'kube-state-metrics'
  metrics_path: /metrics
  static_configs:
    - targets: ['kube-state-metrics.kube-system.svc.cluster.local:8080']
```

1. **blackbox-http**

```yaml
- job_name: 'blackbox-http'
  metrics_path: /probe
  params:
    module: [http_2xx]
  static_configs:
    - targets:
        - http://prometheus.monitoring.svc.cluster.local:9090
        - http://grafana.monitoring.svc.cluster.local:4040
        - http://promtail.monitoring.svc.cluster.local:9080
        - http://loki.monitoring.svc.cluster.local:3100
  relabel_configs:
    - source_labels: [__address__]
      target_label: __param_target
    - source_labels: [__param_target]
      target_label: instance
    - target_label: __address__
      replacement: blackbox-exporter.monitoring.svc.cluster.local:9115
```

1. **node-exporter-job**

```yaml
- job_name: 'node-exporter-job'
  kubernetes_sd_configs:
    - role: endpoints
  metrics_path: /metrics
  relabel_configs:
    - source_labels: [__meta_kubernetes_namespace]
      action: keep
      regex: monitoring
    - source_labels: [__meta_kubernetes_service_name]
      action: keep
      regex: node-exporter
    - source_labels: [__meta_kubernetes_endpoint_port_name]
      action: keep
      regex: http
    - target_label: job
      replacement: node-exporter
```

---

### ➕ Adding New Scrape Jobs

To add a new job (for example, a service exposing metrics via `/metrics` endpoint), **only the new block needs to be appended**. No other changes are required.

### Example: Adding Custom Metrics Scraping

```yaml
- job_name: 'backend-prod-nginx'
  kubernetes_sd_configs:
    - role: endpoints
  metrics_path: /v1/metrics
  relabel_configs:
    - source_labels: [__meta_kubernetes_namespace]
      action: keep
      regex: salemate
    - source_labels: [__meta_kubernetes_service_name]
      action: keep
      regex: backend-prod
    - source_labels: [__meta_kubernetes_endpoint_port_name]
      action: keep
      regex: nginx-metrics
    - target_label: job
      replacement: nginx-sidecar
```

> 📝 Note: This should be added to the scrape_configs: array inside prometheus.yml within the prometheus-config ConfigMap.
>

### Deployment: `prometheus`

Defines the Prometheus server deployment.

### ➤ Configuration

- **Replicas:** 1
- **Node Affinity:** Schedules only on nodes labeled with:
    
    ```
    node-role=monitoring
    ```
    
- **Toleration:** Only schedules on nodes tolerating:
    
    ```
    key=monitoring-only, effect=NoSchedule
    ```
    

### ➤ Volumes

- `alert-rules`: Mounts `prometheus-alert-rules` ConfigMap.
- `prometheus-config`: Mounts `prometheus-config` ConfigMap.
- `prometheus-data`: Mounts a PersistentVolumeClaim `prometheus-pvc`.

---

### Service: `prometheus`

Exposes Prometheus via a LoadBalancer service.

### ➤ Configuration

- **Port:** 9090
- **Target Port:** 9090
- **Type:** `LoadBalancer` for external access

---

### PersistentVolumeClaim: `prometheus-pvc`

Defines the storage for Prometheus metrics data.

### ➤ Key Attributes:

- **Volume Name:** `prometheus-pv`
- **StorageClass:** `local-path-expandable`
- **Access Mode:** `ReadWriteOnce`
