## Blackbox Exporter

### 📌 Purpose

The **Blackbox Exporter** is deployed to monitor the availability of **internal k3s services** using HTTP endpoints via their Kubernetes `svc` DNS names. It enables **external probing** (from the perspective of Prometheus) without requiring direct instrumentation in the monitored services.

Use cases include:

- Validating that HTTP endpoints are reachable within the cluster.
- Confirming TCP-level reachability for services (if extended to TCP modules).
- Supporting synthetic monitoring for both uptime and basic health via probe modules.

---

### Deployment Summary

- **Namespace**: `monitoring`
- **Deployment name**: `blackbox-exporter`
- **Replica count**: `1` (sufficient for internal probing)
- **Node affinity**: Dedicated to the monitoring node via `node-role=monitoring`
- **Toleration**: Supports taints using `monitoring-only=true:NoSchedule`
- **Config source**: `blackbox-exporter-config` (ConfigMap)
- **Service type**: `LoadBalancer` (can be switched to ClusterIP if external access isn't needed)

---

### Configuration

- The exporter uses `blackbox.yml` as its configuration file (mounted from a `ConfigMap`).
- This file defines two **probe modules**:
    
    ```yaml
    modules:
      http_2xx:
        prober: http
        ...
      tcp_connect:
        prober: tcp
        ...
    ```
    

### 🔒 DO NOT modify the ConfigMap or its content:

- It's intentionally scoped and optimized for internal `svc`based endpoints.
- Changing these settings may break the internal HTTP/HTTPS probing and impact alerting rules in Prometheus.

---

### Probing Details

- The `http_2xx` module:
    - Validates HTTP/1.1 and HTTP/2 endpoints.
    - Accepts both SSL and non-SSL endpoints.
    - Uses method `GET` with 5s timeout.
    - Targets IPv4.
- The `tcp_connect` module:
    - Performs TCP connectivity checks (if needed later).
    - Also prefers IPv4.

These modules are directly referenced in the Prometheus scrape configuration.

---

### 📍 Node Placement Logic

This deployment is **restricted to a specific node** labeled with:

```yaml
node-role: monitoring
```

and tolerates this taint:

```yaml
monitoring-only=true:NoSchedule
```

> Ensure your monitoring node is tainted and labeled accordingly:
> 

```bash
kubectl label node monitoring node-role=monitoring
kubectl taint node monitoring monitoring-only=true:NoSchedule
```

### Notes

- Ensure that all internal services you wish to monitor expose an HTTP endpoint and are resolvable via Kubernetes DNS (e.g., `http://my-service.namespace.svc.cluster.local`).
- Prometheus should be configured with an appropriate scrape job using `blackbox-exporter` as the target and setting the `module` parameter.
- The current config is well-tested and production-safe — **do not alter it unless absolutely necessary.**
